# Ch19: Evaluation（评估）总结

## 核心思想

评估是 Agent 系统持续优化的基础。没有评估就没有改进。本章介绍两种评估层次：

1. **基础评估**（准确率/延迟/Token）—— 告诉你「快不快、省不省」
2. **LLM-as-Judge** —— 告诉你「好不好」

## 代码结构

### 第1步：导入依赖
- `langchain_openai.ChatOpenAI` + MiMo 模型

### 第2步：基础评估（纯 Python，不依赖 LLM）
- **精确匹配**：`agent_output == expected_output` → 1.0 或 0.0
- **模糊匹配**：`expected_output in agent_output` → 包含即正确
- **延迟测量**：`time.perf_counter()` 测量函数执行时间（毫秒）
- **Token 监控**：`LLMInteractionMonitor` 类，用 `split()` 粗略估算 token 数

### 第3步：LLM-as-Judge
- 定义评分标准（Rubric）：清晰度、中立性、相关性、完整性，每项 1-5 分
- `llm_as_judge()` 函数：发送评估 prompt → 解析 JSON 输出
- 支持 markdown 代码块中的 JSON 提取

### 第4步：测试 LLM-as-Judge
- 测试案例 1：高质量回答（准确、完整）→ 期望高分
- 测试案例 2：有偏见的回答（引导性语言）→ 期望中等分
- 测试案例 3：模糊的回答（信息不足）→ 期望低分

### 第5步：批量评估 + 结果汇总
- 遍历多个测试案例，收集评估结果
- 生成汇总报告（总分 + 各维度分数 + 理由 + 建议）

## 重难点

### 重点
- **LLM-as-Judge 的 Rubric 设计**：评分标准直接影响评估质量，需要根据业务场景定制
- **JSON 输出解析**：LLM 输出的 JSON 可能不规范，需要容错处理（markdown 代码块提取）

### 难点
- **评估一致性**：LLM 评估可能因 temperature、prompt 微调而产生不一致结果
- **Rubric 泛化**：通用评分标准可能不适用于特定领域，需要领域专家参与设计

## 与其他模式的对比

| 模式 | 关注点 | 评估角度 |
|------|--------|---------|
| Ch4 Reflection | 自我改进 | Agent 自评，无外部标准 |
| Ch11 Goal Setting | 目标达成 | 基于预设目标的二元判断 |
| Ch19 Evaluation | 质量度量 | 多维度量化评分 + 反馈 |

## 评估方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 精确匹配 | 简单、无成本 | 无法处理近似正确 | 事实性问答 |
| 模糊匹配 | 容忍格式差异 | 仍不够智能 | 关键词提取 |
| 延迟测量 | 精确量化性能 | 不评估质量 | 性能优化 |
| Token 监控 | 控制成本 | 不评估质量 | 成本管理 |
| LLM-as-Judge | 多维度、有反馈 | 有成本、可能不一致 | 开放式回答评估 |

## 生产环境建议

- **基础评估实时监控**：延迟、Token、错误率
- **LLM-as-Judge 定期抽检**：每天/每周抽样评估一批回答
- **评估标准定制化**：不要照搬通用模板，根据业务场景设计 Rubric
- **人工校准**：定期用人工评估校准 LLM-as-Judge 的准确性
