# Ch18: Guardrails（护栏/安全）总结

## 核心思想

为 Agent 系统添加多层安全护栏，防止危险操作。生产环境的 Agent 必须有护栏。

## 三层护栏体系

```
用户输入 → [第1层] 输入验证（Pydantic）
         → [第2层] 内容安全（LLM 过滤）
         → Agent 处理
         → [第3层] 输出检查（规则引擎）
         → 返回结果
```

## 代码结构

### 第1步：导入依赖
- `pydantic` 用于数据验证
- `langchain_openai.ChatOpenAI` + MiMo 模型

### 第2步：第1层护栏 — 输入验证（Pydantic）
- `UserQuery` 模型：定义输入格式（类型、长度、正则匹配）
- `validate_input()` 函数：自动验证并返回详细错误信息
- 验证规则：query 长度 1-1000、user_id 格式 USER-XXXX、priority 枚举值

### 第3步：第2层护栏 — 内容安全过滤（LLM）
- `SAFETY_PROMPT`：定义安全检查规则
- `check_content_safety()` 函数：用 LLM 判断输入是否安全
- 检测项：Prompt Injection、危险操作、敏感信息

### 第4步：第3层护栏 — 输出检查（规则引擎）
- `OutputChecker` 类：可扩展的检查器，支持自定义规则
- 内置规则：非空检查、长度限制、隐私检查（手机号）、安全词检查
- `add_check()` 方法：动态添加新规则

### 第5步：整合三层护栏
- `safe_process()` 函数：完整的安全处理流程
- 任何一层拦截都会返回错误信息
- 全部通过才执行 Agent 处理

## 重难点

### 重点
- **多层防御**：不要只依赖一层护栏
- **快速失败**：能在输入阶段拦截的不要等到输出阶段
- **确定性优先**：能用规则判断的不要用 LLM

### 难点
- **误判处理**：LLM 安全检查可能误判，需要 fallback
- **性能权衡**：每层护栏都有延迟，需要平衡安全和性能
- **规则维护**：安全规则需要持续更新

## 各层护栏对比

| 层次 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| 输入验证 | Pydantic | 快速、确定性 | 只能检查格式 |
| 内容安全 | LLM 判断 | 理解语义 | 有成本、可能误判 |
| 输出检查 | 规则引擎 | 快速、可定制 | 需要预定义规则 |

## 与其他模式的关系

| 模式 | 关注点 | 安全角度 |
|------|--------|---------|
| Ch12 Exception Handling | 错误恢复 | 事后处理 |
| Ch13 Human-in-the-Loop | 人工审核 | 事中确认 |
| Ch18 Guardrails | 安全防护 | 事前拦截 |

## 实际应用场景

- 金融 Agent：金额验证、风险评估、合规检查
- 医疗 Agent：症状验证、用药安全、隐私保护
- 客服 Agent：情绪检测、敏感词过滤、升级判断
- 代码 Agent：代码安全扫描、依赖漏洞检查
